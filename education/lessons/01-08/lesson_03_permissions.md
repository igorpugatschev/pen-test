# Занятие 3: Права доступа в Linux

## Теория

В Linux каждый файл и директория имеют права доступа, определяющие кто может читать, записывать и выполнять их.

> **Примечание для macOS (M2, 8GB):** Для запуска Kali Linux на Mac с процессором M2 используйте UTM или Parallels Desktop вместо VirtualBox. Выделяйте виртуалке не более 3-4GB RAM при 8GB на хосте. Установка пакетов в Kali выполняется через `sudo apt install`, для нативных инструментов macOS используйте `brew install`.

### Типы прав доступа

Права обозначаются тремя группами по три бита: `rwx` (read, write, execute)

```
-rwxr-xr--  1 user group  4096 file.txt
↑  ↑↑↑ ↑↑↑ ↑↑↑
│   │││ │││ │││
│   │││ │││ └└└─ права для "others" (остальные)
│   │││ ││└──── права для "group" (группа)
│   │││ └└───── права для "owner" (владелец)
│   └└└─────── тип файла и права
└───────────── тип файла (- обычный, d директория, l ссылка)
```

### Числовое представление (octal)

| Бит | Значение | Число |
|-----|----------|-------|
| r   | чтение   | 4     |
| w   | запись   | 2     |
| x   | выполнение | 1   |

Примеры:
- `755` = rwxr-xr-x (владелец всё, остальные только чтение и выполнение)
- `644` = rw-r--r-- (владелец читает/пишет, остальные только читают)
- `700` = rwx------ (только владелец имеет доступ)

### Ключевые команды

| Команда | Назначение |
|---------|-----------|
| `chmod` | Изменить права доступа |
| `chown` | Изменить владельца и группу |
| `sudo` | Выполнить команду от имени root |
| `su` | Переключиться на другого пользователя |
| `id` | Показать UID, GID, группы |
| `whoami` | Показать текущего пользователя |

### SUID, SGID, Sticky bit

- **SUID** (4000) — программа выполняется с правами владельца (например, `/usr/bin/passwd`)
- **SGID** (2000) — программа выполняется с правами группы
- **Sticky bit** (1000) — в директории (например `/tmp`) файлы может удалить только владелец

## Практическое занятие

### Шаг 1: Проверка текущего пользователя и прав

```bash
# Кто я?
whoami
```
**Пример вывода `whoami`:**
```text
user
```

```bash
# Мои права и группы
id
```
**Пример вывода `id`:**
```text
uid=1000(user) gid=1000(user) groups=1000(user),27(sudo)
```

```bash
# Список пользователей в системе
cat /etc/passwd
```
**Пример вывода `cat /etc/passwd` (фрагмент):**
```text
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
user:x:1000:1000:user:/home/user:/bin/bash
```

```bash
# Список групп
cat /etc/group
```

### Шаг 2: Просмотр и изменение прав (chmod)

```bash
cd ~/terminal_practice

# Создать тестовый файл
echo "secret data" > secret.txt

# Посмотреть права
ls -l secret.txt
```
**Пример вывода `ls -l secret.txt`:**
```text
-rw-r--r-- 1 user user 12 Jan 10 10:00 secret.txt
```

```bash
# Убрать все права для others
chmod o-rwx secret.txt
ls -l secret.txt
```
**Пример вывода:**
```text
-rw-r----- 1 user user 12 Jan 10 10:00 secret.txt
```

```bash
# Дать все права владельцу, никаких для остальных
chmod 700 secret.txt
ls -l secret.txt
```
**Пример вывода:**
```text
-rwx------ 1 user user 12 Jan 10 10:00 secret.txt
```

```bash
# Сделать файл исполняемым
echo '#!/bin/bash' > script.sh
echo 'echo "Hello"' >> script.sh
chmod +x script.sh
ls -l script.sh
```
**Пример вывода:**
```text
-rwxr-xr-x 1 user user 24 Jan 10 10:01 script.sh
```

```bash
./script.sh
```
**Пример вывода:**
```text
Hello
```

### Шаг 3: Изменение владельца (chown)

```bash
# Посмотреть текущего владельца
ls -l secret.txt

# Создать тестового пользователя (если нет)
sudo useradd testuser

# Изменить владельца файла
sudo chown testuser secret.txt
ls -l secret.txt
```
**Пример вывода:**
```text
-rwx------ 1 testuser user 12 Jan 10 10:00 secret.txt
```

```bash
# Изменить владельца и группу
sudo chown testuser:testuser secret.txt
ls -l secret.txt

# Вернуть себе
sudo chown $USER:$USER secret.txt
```

### Шаг 4: Работа с sudo и su

```bash
# Выполнить команду от root
sudo cat /etc/shadow
```
**Пример вывода `sudo cat /etc/shadow` (фрагмент):**
```text
root:$6$abc123$...:19000:0:99999:7:::
user:$6$def456$...:19000:0:99999:7:::
```

```bash
# Переключиться на root (нужен пароль root)
sudo -i
# или
su -
# Выйти из root
exit

# Посмотреть, кто может использовать sudo
cat /etc/sudoers
sudo cat /etc/sudoers
```

### Шаг 5: SUID и поиск уязвимостей

> **Примечание:** Команда `find` подробно изучается в Уроке 6. Кратко: `find` ищет файлы по файловой системе по заданным критериям.

```bash
# Найти все файлы с SUID-битом (потенциальные векторы повышения привилегий)
find / -perm -4000 2>/dev/null
```
**Пример вывода:**
```text
/usr/bin/passwd
/usr/bin/sudo
/usr/bin/su
```

```bash
# Посмотреть SUID у конкретного файла
ls -l /usr/bin/passwd
```
**Пример вывода:**
```text
-rwsr-xr-x 1 root root 68208 Jan 10 09:00 /usr/bin/passwd
```

```bash
# Найти файлы с правами записи для всех
find / -perm -002 2>/dev/null | head -20
```

## Задачи для самостоятельного выполнения

1. Создайте файл `private.txt` и установите права `600` (только владелец может читать и писать). Проверьте, что другой пользователь (или от имени другого пользователя через `su`) не может прочитать файл.

2. Создайте директорию `shared/` и установите права `775`. Создайте внутри файл `team.txt` с правами `664`. Объясните, почему выбраны такие права.

3. Найдите в системе все файлы с установленным SUID-битом. Сохраните список в `~/suid_files.txt`. *Примечание: используется команда `find`, которая подробно изучается в Уроке 6. Краткий синтаксис: `find / -perm -4000 2>/dev/null`.* Исследуйте, какие из них могут быть использованы для повышения привилегий (nmap, find, vim с SUID — критично).

4. Создайте скрипт `backup.sh`, который архивирует директорию `~/practice` в `~/backup.tar.gz`. Сделайте скрипт исполняемым только для владельца. Запустите скрипт через `sudo` и убедитесь, что архив создан. *Примечание: создание bash-скриптов подробно изучается в Уроке 7. Кратко: скрипт начинается с `#!/bin/bash` и делается исполняемым через `chmod +x`.*

5. Исследуйте файл `/etc/sudoers` и файлы в `/etc/sudoers.d/`. Узнайте, как дать пользователю право запускать конкретную команду без ввода пароля. Напишите пример строки для sudoers.

## Частые ошибки

1. **Неправильный подсчет прав chmod.** Часто забывают, что r=4, w=2, x=1. Например, 777 дает все права всем, что опасно для системных файлов.
2. **Использование chown без sudo.** Обычный пользователь не может менять владельца файлов, принадлежащих root.
3. **Случайная установка SUID-бита на пользовательские скрипты.** Это позволит любому запускать скрипт с правами владельца, создавая уязвимость.
4. **Путаница между `su` и `sudo`.** `su` переключает пользователя (требует пароль целевого пользователя), `sudo` выполняет команду от root (требует пароль текущего пользователя).
