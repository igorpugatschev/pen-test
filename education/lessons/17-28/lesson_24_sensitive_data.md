# Занятие 24. Sensitive Data Exposure: Перехват трафика, поиск секретов

## Теория

**Sensitive Data Exposure (A02:2021)** — это уязвимость, при которой приложение неправильно защищает конфиденциальные данные: пароли, данные карт, персональные данные (PII).

### Как происходит утечка

1. **Передача по HTTP**: Данные передаются в открытом виде, их можно перехватить (Sniffing)
2. **Слабое шифрование**: Использование устаревших алгоритмов (MD5, SHA1) или отсутствие соли
3. **Хранение в открытом виде**: Пароли, ключи API, токены лежат в БД как есть
4. **Неправильная конфигурация**: Доступ к файлам .env, backup, конфигам без авторизации
5. **Утечка в ответах сервера**: Сервер возвращает лишние данные (номера карт, пароли в ответе)

### Что относится к чувствительным данным

- Учетные данные (пароли, токены сессий)
- Финансовая информация (номера карт, счета)
- Персональные данные (паспорт, СНИЛС, адрес, телефон)
- Ключи шифрования, API ключи, секреты приложения

### Механизм атаки

1. Атакующий перехватывает трафик (через MITM, например, в публичном Wi-Fi)
2. Если трафик не защищен HTTPS, он видит все данные
3. Если HTTPS есть, но сертификат самоподписанный или слабый — можно подделать
4. Атакующий получает доступ к файлам с конфигом (`.env`, `config.php`)

### Инструменты для перехвата

- **Wireshark**: Анализатор сетевого трафика
- **Burp Suite**: Прокси для перехвата HTTP/HTTPS
- **tcpdump**: Консольный сниффер
- **mitmproxy**: Прокси для MITM атак

## Практическое занятие

### Перехват трафика в DVWA (HTTP)

DVWA по умолчанию работает по HTTP (порт 80) — трафик не шифруется.

**Шаг 1: Настройка Wireshark**
1. Запустите Wireshark
2. Выберите интерфейс `lo0` (loopback, так как Docker на локалке)
3. В фильтре введите: `http and contains "password"`

**Шаг 2: Выполнение входа в DVWA**
1. Откройте http://localhost
2. Введите `admin` / `password`
3. Посмотрите в Wireshark — найдите POST-запрос `/login.php`

**Шаг 3: Анализ перехваченного запроса**
В Wireshark найдите пакет с POST /login.php:
```
POST /login.php HTTP/1.1
Host: localhost
Content-Type: application/x-www-form-urlencoded
Content-Length: 45

username=admin&password=password&Login=Login
```
Пароль передается в открытом виде!

**Шаг 4: Поиск секретов в файлах (bWAPP)**
Откройте bWAPP. Иногда конфиг лежит в `/config.php` или `/admin/config.php`.
Попробуйте открыть: http://localhost/config.inc.php — если доступно, увидите настройки БД.

**Шаг 5: Анализ через Burp Suite**
1. Настройте браузер на прокси 127.0.0.1:8080
2. Установите CA-сертификат Burp (для HTTPS)
3. Войдите в DVWA — запрос появится в Proxy → HTTP history
4. Посмотрите вкладку **Headers** и **Request** — видны ли пароли?

### Поиск утечек в ответах сервера

В DVWA перейдите в **View Source** любой страницы. Иногда комментарии содержат секреты:
```html
<!-- TODO: change default password for admin -->
<!-- DB password: password123 -->
```

### Скриншоты для отчета

1. **Скриншот 1**: Wireshark — перехваченный POST-запрос с паролем в открытом виде
2. **Скриншот 2**: Burp Suite — запрос логина, видны имя и пароль
3. **Скриншот 3**: Утечка в HTML-комментариях (View Source)

## Задачи для самостоятельного выполнения

1. **Перехват с Wireshark**: Используя Wireshark, перехватите трафик при входе в bWAPP (bee/bug). Найдите в пакете данные логина и пароля. Сделайте скриншот пакета с видимыми учетными данными.

2. **Поиск конфигов**: В bWAPP попробуйте открыть следующие пути (если доступны):
   - `/config.php`
   - `/install.php`
   - `/admin/`
   - `/backup/`
   
   Опишите, какие файлы доступны без авторизации, какие данные утекли.

3. **Хеши паролей**: В DVWA (Low) выполните SQL Injection, чтобы получить хеши паролей пользователей. Определите, какой алгоритм используется (MD5, SHA1 и т.д.). Попробуйте расшифровать хеш пользователя `gordonb` через crackstation.net.

4. **HTTPS vs HTTP**: Объясните, почему передача паролей по HTTP опасна, даже если пароль хешируется на клиенте (JavaScript). Почему HTTPS обязателен для всех страниц с аутентификацией?

5. **Поиск секретов в WebGoat**: В WebGoat перейдите в раздел **Sensitive Data Exposure**. Выполните задания (минимум 2). Опишите, какие уязвимости вы нашли, какие данные утекли. Сделайте скриншот выполненных заданий.
