# Занятие 17b. OWASP A04: Insecure Design — Небезопасный дизайн

## Теория

**A04:2021 – Insecure Design** — это уязвимости, заложенные на этапе проектирования архитектуры приложения. Их нельзя исправить простым патчем — нужно менять архитектуру.

### Отличие Insecure Design от Insecure Implementation

| Характеристика | Insecure Design | Insecure Implementation |
|----------------|-----------------|-------------------------|
| Где ошибка | В архитектуре | В коде |
| Когда возникает | На этапе проектирования | На этапе разработки |
| Как исправить | Переделать архитектуру | Исправить код |
| Пример | Нет ограничения попыток входа | Слабое шифрование пароля |

### Примеры небезопасного дизайна

1. **Отсутствие rate limiting** — нет защиты от брутфорса
2. **Отсутствие workflow для смены пароля** — восстановление без подтверждения
3. **Прямой доступ к объектам без проверки прав** — IDOR по дизайну
4. **Отсутствие аудита** — нет логирования критических действий
5. **Доверие клиентскому вводу** — важные решения принимаются на клиенте

### Почему это важно

Insecure Design попал в топ-10 OWASP 2021 на 4-е место, потому что:
- Многие уязвимости (IDOR, Missing Function Level Access Control) — это проблемы дизайна
- Патчи не помогают, если архитектура неверна
- Threat Modeling часто игнорируется

### Пример уязвимого дизайна

**Сценарий**: Функция восстановления пароля отправляет ссылку на email без токена:
```
GET /reset-password?email=user@example.com
```
Сервер сразу меняет пароль и отправляет новый на почту. Атакующий может перебирать email-ы и сбрасывать пароли всем пользователям.

**Правильный дизайн**: Генерируется токен, сохраняется в БД, отправляется на email. Смена только после перехода по ссылке с валидным токеном.

## Практическое занятие

### Анализ дизайна DVWA

**Шаг 1: Изучение механизма смены пароля**
1. В DVWA перейдите в **CSRF** (Low)
2. Посмотрите форму смены пароля
3. Откройте Burp → Proxy, перехватите запрос смены пароля

Пример запроса:
```
POST /vulnerabilities/csrf/ HTTP/1.1
Host: localhost
Cookie: PHPSESSID=abc123; security=low
Content-Type: application/x-www-form-urlencoded

password_new=hacked&password_conf=hacked&Change=Change
```

**Шаг 2: Проблема дизайна**
- Нет CSRF-токена (только на уровне Low, на Medium появляется)
- Нет подтверждения через старый пароль
- Нет уведомления на email о смене пароля

**Шаг 3: Анализ отсутствия rate limiting**
1. Включите Burp Proxy
2. Попробуйте войти с неправильным паролем 10 раз подряд
3. DVWA не блокирует — это проблема дизайна

Результат в Burp HTTP history:
```
POST /login.php
...
username=admin&password=wrong1&Login=Login
→ Response: 200 OK (Login failed)

POST /login.php
...
username=admin&password=wrong2&Login=Login
→ Response: 200 OK (Login failed)
```
Никакой задержки, никакой блокировки.

### Поиск проблем дизайна в bWAPP

**Шаг 4: Прямой доступ к админке**
1. В bWAPP войдите как `bee` / `bug`
2. Попробуйте открыть `/admin/` напрямую
3. Если доступно без проверки прав — это проблема дизайна (Missing Function Level Access Control)

---

## Частые ошибки

1. **Путать с реализацией** — если код не проверяет права, это implementation; если архитектура не предусматривает проверку — design
2. **Игнорировать Threat Modeling** — начинать кодить без анализа угроз
3. **Доверять клиенту** — делать важные проверки на стороне браузера

## Адаптация под macOS (M2)

```bash
# Установка инструментов для анализа архитектуры
brew install --cask drawio  # для создания диаграмм архитектуры

# Тестирование rate limiting через curl
for i in {1..10}; do
  curl -X POST http://192.168.0.x/login.php \
    -d "username=admin&password=wrong$i&Login=Login" \
    -c cookies.txt -b cookies.txt -s -o /dev/null -w "%{http_code}\n"
done
```

## Адаптация под macOS (M2, 8GB)

- Для установки инструментов используйте Homebrew: `brew install <tool>`

- Используйте UTM вместо VirtualBox (лучшая поддержка ARM)
- Docker работает нативно на M2: `docker pull <image>`

## Задачи для самостоятельного выполнения

1. **Анализ дизайна DVWA**: Изучите функцию смены пароля в DVWA (CSRF). Опишите в отчете: какие проверки отсутствуют, почему это проблема дизайна, как должно быть правильно.

2. **Поиск IDOR**: В DVWA перейдите в **SQL Injection (Blind)**. Попробуйте изменить параметр `id` в URL на другие значения. Если без проверки прав вы видите чужие данные — опишите это как проблему дизайна.

3. **Threat Modeling**: Выберите простую функцию (например, логин в bWAPP). Составьте список угроз (минимум 3): что может пойти не так, если не заложить защиту на этапе дизайна.

4. **Rate Limiting тест**: Используя Burp Intruder, отправьте 20 запросов логина в DVWA (Low). Посмотрите, появляется ли задержка или блокировка. Сделайте скриншот результатов в Intruder.

5. **Сравнение уровней**: Сравните уровни Low и Medium в DVWA с точки зрения дизайна. Появляется ли на Medium защита от CSRF (токены)? Изменилась ли архитектура или просто добавилась проверка в коде?
