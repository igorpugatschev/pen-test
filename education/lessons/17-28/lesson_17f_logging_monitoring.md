# Занятие 17f. OWASP A09: Security Logging and Monitoring Failures — Отсутствие логирования и мониторинга

## Теория

**A09:2021 – Security Logging and Monitoring Failures** — отсутствие или неэффективное логирование атак и событий безопасности, а также отсутствие мониторинга для своевременного обнаружения взлома.

### Почему это критично

1. **Атака не обнаруживается** — атакующий может часами сканировать сайт, никто не узнает
2. **Нет данных для расследования** — после взлома невозможно понять, что произошло
3. **Задержка обнаружения** — по статистике, взломы обнаруживаются через 200+ дней

### Что должно логироваться

| Событие | Важность |
|---------|----------|
| Неудачные попытки входа | Высокая (брутфорс) |
| Успешный вход | Средняя (аномалии) |
| Смена пароля | Высокая |
| Доступ к админке | Критическая |
| Ошибки доступа (403, 401) | Средняя |
| SQL Injection, XSS пейлоады | Критическая |

### Чего НЕ должно быть в логах

- Пароли (даже временно)
- Токены сессий
- Персональные данные (PII) в открытом виде
- Данные банковских карт

### Пример плохого логирования

```
[2024-01-15 10:30:22] Login failed for user admin
```
Нет IP-адреса, нет количества попыток, нет деталей.

### Пример хорошего логирования

```
[2024-01-15 10:30:22] LOGIN_FAILED user=admin ip=192.168.1.100 
attempt=5/10 user_agent=Mozilla/5.0 referer=http://site.com/login
```

## Практическое занятие

### Анализ логирования в DVWA

**Шаг 1: Попытка найти логи**
1. Войдите в DVWA как admin
2. Попробуйте неправильный пароль 3 раза
3. Проверьте, есть ли в интерфейсе раздел с логами

В DVWA (Low/Medium) **нет логов атак** — это проблема A09.

**Шаг 2: Проверка логов веб-сервера (через Docker)**
```bash
docker ps  # найдите ID контейнера DVWA
docker logs <container_id>
```

Пример вывода (Apache логи):
```
172.17.0.1 - - [15/Jan/2024:10:30:22 +0000] "POST /login.php HTTP/1.1" 302 -
172.17.0.1 - - [15/Jan/2024:10:30:25 +0000] "GET /vulnerabilities/sqli/?id=1 HTTP/1.1" 200 1234
172.17.0.1 - - [15/Jan/2024:10:30:30 +0000] "GET /vulnerabilities/xss_r/?name=<script>alert(1)</script> HTTP/1.1" 200 567
```

XSS-пейлоад в логе! Но никто не мониторит это.

**Шаг 3: Проверка отсутствия алертов**
1. Включите Burp Proxy
2. Отправьте 20 запросов с неверным паролем (брутфорс)
3. Посмотрите, появится ли блокировка или уведомление

Результат: DVWA продолжает принимать запросы — нет мониторинга.

**Шаг 4: Поиск логов в bWAPP**
Откройте bWAPP, попробуйте выполнить SQL Injection:
```
http://192.168.0.x/sqli_1.php?title=1' OR '1'='1
```

Проверьте, логируется ли это. В типичной конфигурации bWAPP — нет.

## Частые ошибки

1. **Логировать пароли** — они попадают в логи в открытом виде
2. **Не мониторить логи** — логи пишутся "для галочки", никто их не читает
3. **Писать слишком мало информации** — только факт события без IP, времени, user-agent
4. **Писать слишком много** — логи разрастаются, важное тонет в шуме

## Адаптация под macOS (M2)

```bash
# Просмотр системных логов (работает на M2)
log show --predicate 'eventMessage contains "Failed"' --last 1h

# Установка SIEM для тестирования (через Docker на M2)
docker run -d -p 5601:5601 -p 9200:9200 -p 5044:5044 \
  sebp/elk  # Elasticsearch + Logstash + Kibana

# Отправка логов в syslog
logger -t "SECURITY" "Failed login attempt from 192.168.1.100 for user admin"
```

## Адаптация под macOS (M2, 8GB)

- Для установки инструментов используйте Homebrew: `brew install <tool>`
- На MacBook Air M2 (8GB) запускайте VM с памятью не более 3-4GB
- Используйте UTM вместо VirtualBox (лучшая поддержка ARM)
- Docker работает нативно на M2: `docker pull <image>`
- Для VPN используйте Tunnelblick (OpenVPN) или официальные клиенты
- Для Python используйте `pip3 install` вместо `pip install`

1. **Анализ логирования DVWA**: Попробуйте выполнить 5 неудачных попыток входа в DVWA. Посмотрите логи Docker контейнера (`docker logs`). Есть ли там информация об атаках? Чего не хватает в логах?
