# Занятие 17. OWASP Top 10: Введение в 10 категорий уязвимостей

## Теория

**OWASP Top 10** — это стандартный документ, публикуемый организацией OWASP (Open Web Application Security Project), который перечисляет 10 наиболее критичных рисков безопасности веб-приложений. Документ обновляется раз в несколько лет на основе статистики реальных взломов.

### Актуальный список OWASP Top 10 (2021)

1. **A01:2021 – Broken Access Control** — нарушение контроля доступа. Позволяет атакующему получить доступ к данным или функциям, которые ему не принадлежат.
2. **A02:2021 – Cryptographic Failures** — криптографические ошибки. Утечка чувствительных данных из-за слабого шифрования или его отсутствия.
3. **A03:2021 – Injection** — инъекции (SQL, NoSQL, LDAP, OS команд). Недоверенный ввод попадает в интерпретатор без проверки.
4. **A04:2021 – Insecure Design** — небезопасный дизайн. Ошибки архитектуры, закладывающие уязвимости на этапе проектирования.
5. **A05:2021 – Security Misconfiguration** — неправильная конфигурация безопасности. Открытые порты, дефолтные пароли, лишние сервисы.
6. **A06:2021 – Vulnerable and Outdated Components** — уязвимые и устаревшие компоненты. Использование старых версий библиотек с известными CVE.
7. **A07:2021 – Identification and Authentication Failures** — ошибки идентификации и аутентификации. Слабые пароли, отсутствие MFA, плохая обработка сессий.
8. **A08:2021 – Software and Data Integrity Failures** — нарушение целостности ПО и данных. Небезопасные обновления, десериализация без проверки.
9. **A09:2021 – Security Logging and Monitoring Failures** — отсутствие логирования и мониторинга. Не позволяет обнаружить атаку вовремя.
10. **A10:2021 – Server-Side Request Forgery (SSRF)** — подделка запросов на стороне сервера. Сервер делает запросы по указанию атакующего.

### Как формируется список

OWASP собирает данные от компаний, занимающихся безопасностью, bug bounty платформ и статистики взломов. Каждая уязвимость оценивается по трем критериям:
- **Exploitability** — насколько легко её эксплуатировать
- **Prevalence** — насколько часто встречается
- **Detectability** — насколько легко обнаружить
- **Impact** — насколько велик ущерб

### CVSS (Common Vulnerability Scoring System)

Каждая уязвимость имеет оценку от 0 до 10:
- 0.1–3.9: Low
- 4.0–6.9: Medium
- 7.0–8.9: High
- 9.0–10.0: Critical

## Практическое занятие

### Установка DVWA (Damn Vulnerable Web Application)

DVWA — это специально уязвимое веб-приложение для обучения пентестингу.

**Через Docker (рекомендуется):**
```bash
docker run --rm -it -p 80:80 vulnerables/web-dvwa
```

**Через Docker Compose:**
```yaml
version: '3'
services:
  dvwa:
    image: vulnerables/web-dvwa
    ports:
      - "80:80"
    environment:
      - MYSQL_PASS=password
```

После запуска откройте http://192.168.0.x/setup.php и нажмите **Create / Reset Database**.

**Данные для входа по умолчанию:**
- Login: `admin`
- Password: `password`

### Установка bWAPP

bWAPP (Buggy Web Application) — еще одно уязвимое приложение.

```bash
docker run -d -p 80:80 raesene/bwapp
```

Откройте http://192.168.0.x/install.php и нажмите **here to install bWAPP**.

**Данные для входа:**
- Login: `bee`
- Password: `bug`

### Установка WebGoat

WebGoat — обучающая платформа от OWASP с пошаговыми уроками.

```bash
docker run -d -p 8080:8080 webgoat/goatandwolf
```

Откройте http://192.168.0.x80/WebGoat — регистрация не требуется, просто введите имя пользователя.

### Навигация по DVWA

После входа в DVWA:
1. Перейдите в **DVWA Security** — установите уровень **Low** (для начала)
2. В меню слева выберите **SQL Injection** — это первая уязвимость для изучения
3. Попробуйте ввести `1` в поле User ID — приложение вернёт данные пользователя

### Знакомство с интерфейсом

В каждом приложении:
- **DVWA**: меню слева содержит список всех уязвимостей
- **bWAPP**: выпадающий список уязвимостей на главной странице
- **WebGoat**: пошаговые уроки в боковой панели

### Пропущенные категории OWASP Top 10 (подробнее в уроках 17b-17f)

- **A04: Insecure Design** — ошибки архитектуры (урок 17b)
- **A05: Security Misconfiguration** — неправильная конфигурация (урок 17c)
- **A06: Vulnerable and Outdated Components** — устаревшие компоненты (урок 17d)
- **A08: Software and Data Integrity Failures** — нарушение целостности (урок 17e)
- **A09: Security Logging and Monitoring Failures** — отсутствие логирования (урок 17f)

### Пример вывода после установки DVWA

При открытии http://192.168.0.x после запуска Docker:
```
DVWA - Damn Vulnerable Web Application
Login: admin
Password: password
Setup: http://192.168.0.x/setup.php
```

## Адаптация под macOS (M2, 8GB)

- Для установки инструментов используйте Homebrew: `brew install <tool>`
- На MacBook Air M2 (8GB) запускайте VM с памятью не более 3-4GB
- Используйте UTM вместо VirtualBox (лучшая поддержка ARM)
- Docker работает нативно на M2: `docker pull <image>`
- Для VPN используйте Tunnelblick (OpenVPN) или официальные клиенты
- Для Python используйте `pip3 install` вместо `pip install`


## Задачи для самостоятельного выполнения

1. **Установка лаборатории**: Разверните DVWA, bWAPP и WebGoat через Docker. Убедитесь, что все три приложения доступны в браузере. Сделайте скриншот главной страницы каждого приложения.

2. **Изучение OWASP Top 10**: Прочитайте официальный документ OWASP Top 10 2021 на сайте owasp.org. Составьте таблицу из 10 строк: номер, название уязвимости, краткое описание (1 предложение), пример (1 пример из жизни).

3. **Разведка DVWA**: Изучите исходный код DVWA (доступен на GitHub: github.com/digininja/DVWA). Найдите файл, отвечающий за SQL Injection на уровне Low. Какой файл содержит уязвимый код? Сколько строк кода в этом файле?

4. **Настройка уровней безопасности**: В DVWA переключите уровень безопасности с Low на Medium и снова откройте SQL Injection. Попробуйте ввести `1' OR '1'='1`. Что изменилось? Почему запрос больше не работает?

5. **WebGoat первый урок**: В WebGoat пройдите введение (Introduction → WebGoat Introduction). Выполните задание "HTTP Basics". Сделайте скриншот успешного выполнения задания с зелёной галочкой.

### Частые ошибки

1. **Неправильный порт** — Docker может назначить другой порт, проверяйте `docker ps`
2. **Забыли создать БД** — всегда открывайте `/setup.php` перед первым входом
3. **Уровень безопасности** — начинайте с Low, иначе пейлоады не сработают

### Вопросы на понимание

1. Чем отличается A01 (Broken Access Control) от A07 (Identification and Authentication Failures)?
2. Почему Insecure Design (A04) нельзя исправить патчем?
3. Как понять, что компонент уязвим (A06)?
4. Зачем нужно логирование (A09) при обнаружении атаки?
